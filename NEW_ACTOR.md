# Overview

All of our existing scan types use pre-existing Apify Actors. 

We're now going to make our own 'Recent domain registrations scanner' Actor.

This will wrap the CodePunch GTLD Domain Name Acitvity Feed API (V2), which is documented at https://codepunch.com/dnfeed/v2/ (the /added endpoint).


# Requirements

The following should be configurable by users of the Actor
- Their CodePunch API key and API secret (stored securely)*
- The date in yyyymmdd format*
- The date comparison approach (eq, lt, gt, lte, gte)*
- Keywords (the API allows keywords to be pipe-separated and % used for wildcards, but if possible, the actor should hand this in a way that multiple keywords can be defined in a collection and then translated into this format before sending to the API)*
- The TLDs to consider (comma-separated list) (Optional)
- The sort field (choice of date, domain or tld) (Optional)
- The sort order (chose of asc or desc) (Optional)
* = required

The dm param should always be set to data.

Teh Actor should return data in an Apify-friendly format.

We should publish the Actor to Apify so that we can use it soon in our app (no need to actually wire it in yet - we'll do that later).



# Considerations
- The API requires the generation of an access token (using the API key and secret), which is then used to authorise API requests
 - In theory , tokens are IP-scoped and don't expire, but we can't rely on this and we need to consider that Apify's IP addresses may change per request
 - If we're storing tokens in Apify, we should do so in a secure way
 - There are rate limits around this, so we can't assume that we can generate a new auth token per request (fast concurrent requests might contravene rate limits)
